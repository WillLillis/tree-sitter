# Corpus verification handoff: rule_ir vs master, full grammar sweep

You are a Claude instance on a machine that has the full set of
nvim-treesitter grammar repos cloned. Your job: verify that the
`rule_ir` branch of tree-sitter produces **byte-identical generated
output** to master for every grammar in that corpus, and report the
results. You have no other context; everything you need is below.

This file is a temporary working doc on the `rule_ir` branch (same
convention as `FLAT_POOL_INVESTIGATION.md`); it is deleted before any PR.

## Background (one paragraph)

`rule_ir` replaces the grammar generator's internal representation (a
recursive `Rule` box-tree) with a flat pool IR, and layers eight
performance levers on `build_tables` (bitset indexes, interned lookahead
sets, item shrink, memoization, signature-bucketed state merging, inline
action lists). Net effect on the six in-repo fixture grammars: generate is
roughly 2x faster and uses roughly one third of the peak memory, with
parser.c and node-types.json byte-identical to master at every commit.
The six fixtures (c, cpp, python, go, javascript, rust) plus the ~90
test_grammars in `test/fixtures/test_grammars` are the only verification
so far. This sweep is the scale-up: byte-identical across the whole
corpus, or a pinpointed culprit commit for any divergence.

## Rules (hard-won; do not skip)

1. **Never pipe-truncate a generate run.** SIGPIPE from `head`/`grep -m1`
   kills the process before verification completes. Always redirect to
   files and check exit codes.
2. **Do not use `git worktree`** (repo-owner rule). To build the two
   binaries from one clone, use `git archive <rev> | tar -x -C <dir>` and
   build from the exported tree, or two sequential checkouts saving the
   binary between builds.
3. **The branch binary prints `[SPIKE ...]` diagnostic lines to stderr.**
   Filter them (`grep -v '^\[SPIKE'`) before comparing stderr between
   binaries. They are temporary instrumentation, not output.
4. Compare the two binaries' outputs against **each other**, never against
   a repo's checked-in `src/parser.c` (those were generated by arbitrary
   CLI versions and will differ legitimately).
5. Generate from each grammar's checked-in **`src/grammar.json`** (the
   JSON path needs no JS runtime and is exactly what the branch changed).
   Do not regenerate grammar.json from grammar.js.

## Build the two binaries

```sh
REPO=<path to the tree-sitter clone>   # must have rule_ir fetched
WORK=<scratch dir>
cd "$REPO"

git archive 9fc2f486a | (mkdir -p "$WORK/src-master" && tar -x -C "$WORK/src-master")
git archive rule_ir   | (mkdir -p "$WORK/src-branch" && tar -x -C "$WORK/src-branch")

# Shared target dir avoids rebuilding dependencies twice.
CARGO_TARGET_DIR="$WORK/target" cargo build --release -p tree-sitter-cli \
  --manifest-path "$WORK/src-master/Cargo.toml"
cp "$WORK/target/release/tree-sitter" "$WORK/ts-master"

CARGO_TARGET_DIR="$WORK/target" cargo build --release -p tree-sitter-cli \
  --manifest-path "$WORK/src-branch/Cargo.toml"
cp "$WORK/target/release/tree-sitter" "$WORK/ts-branch"
```

Sanity check before the sweep: both binaries on one small grammar, outputs
must be byte-identical (see comparison below). If even that fails, stop
and report; something about the environment differs from expectations.

## The sweep

Grammar discovery: nvim-treesitter clones keep one repo per language, but
several repos contain **multiple grammars in subdirectories** (typescript
has typescript/ and tsx/, markdown has markdown/ and markdown-inline/,
ocaml, php, etc.). Discover grammars as every `src/grammar.json` under
each repo, not one per repo:

```sh
find "$CORPUS" -name grammar.json -path '*/src/*' -not -path '*/node_modules/*'
```

For each discovered `src/grammar.json`:

```sh
name=<derived from path, unique>
for side in master branch; do
  mkdir -p "$WORK/out/$side/$name"
  "$WORK/ts-$side" generate "$gj" -o "$WORK/out/$side/$name" \
    > "$WORK/out/$side/$name.stdout" 2> "$WORK/out/$side/$name.stderr"
  echo $? > "$WORK/out/$side/$name.exit"
done
```

Classify each grammar:

- Both exit 0 and `parser.c` + `node-types.json` byte-identical
  (`cmp`): **IDENTICAL** (the expected outcome).
- Both exit 0, files differ: **OUTPUT_DIFFERS** (a finding; see below).
- Different exit codes: **EXIT_DIFFERS** (a finding).
- Both nonzero, same stderr after filtering `[SPIKE` lines from the
  branch side: **BOTH_FAIL_SAME** (fine; some corpus grammars fail on
  this CLI version, equivalence is what matters).
- Both nonzero, different messages: **BOTH_FAIL_DIFF** (a finding, softer;
  error-message drift matters less than output drift but report it).
- No src/grammar.json found for a repo: **SKIPPED** (list them).

Grammars can be generated in parallel (independent processes); keep any
timing measurements in a separate, sequential phase.

## If you find a divergence

Every campaign commit was individually byte-verified on the six fixtures,
so a single culprit commit exists and `git bisect` will find it. The
interesting range and the lever commits, oldest first:

```
676189b3a parse_grammar: build the pool directly (stage 3 frontend flip)
99f57e64d build_tables: delete the item-op spike counters
b6a2bbc24 build_tables: coincident state lists -> without-word bitset
87e15dbc3 build_tables: stop duplicating kernel item sets
393fe558d minimize: vectorize token_conflicts with precomputed bitsets
49e9b232a build_tables: intern lookahead TokenSets
87b1addec build_tables: shrink ParseItem to a production id
44a4e3b1e build_tables: memoize get_production_id by production id
0bd596e80 minimize: bucket split1 by state signature
0bf687b65 tables: inline single-action parse table entries
```

Bisect recipe (archive-based, no worktrees): `git bisect start rule_ir
9fc2f486a`, and for each candidate rev, `git archive <rev>` to a scratch
dir, build with the shared target dir, generate the failing grammar, `cmp`
against the master output; `git bisect good/bad` accordingly. Earlier
commits (the pool IR conversion series before the list above) are also in
range if the culprit is not a lever; the same recipe applies.

For any OUTPUT_DIFFERS, include in the report: the grammar, the culprit
commit from bisect, and the first ~40 lines of
`diff -u master/parser.c branch/parser.c` so the divergence class is
visible (state renumbering vs action changes vs symbol naming).

## Optional phase 2: corpus timing

Only after correctness is settled. Interleaved min-of-3 per grammar
(`/usr/bin/time -f "%e %M"`, alternate binaries within each round, never
back-to-back same-binary), sequential, machine otherwise idle. Report the
corpus-wide totals (sum of min wall, both sides) and a per-grammar table
for the 10 largest grammars by master wall time. On the six in-repo
fixtures this branch measures 43-57% faster wall and 43-69% lower peak
RSS; corpus-scale confirmation (or refutation) of that range is the value.
